## Общие требования к выполнению задания

0. Вносить изменения в скрипты и файлы с кодом **запрещено**
1. Название проекта должно быть `train_your_dragon`
2. В качестве минимальной версии CMake укажите 3.11
3. В качестве стандартов языков C и C++ укажите 17
4. Свои CMake Modules расположите в папке `/cmake`

> Сейчас в папке `/cmake` лежит файл `.gitkeep` - вспомните *task-git*, зачем это было сделано?
> Что следует сделать с этим файлом, после добавления модулей в папку?

5. Добавьте вывод стандартов языков при запуске CMake.
Сообщения должны быть `C standard 17` и `C++ standard 17`
6. Старайтесь выводить сообщения, которые помогают понять, какие процессы происходят, во время сборки. (Такое еще называют логами сборки)

## Этап 1 (1.5 балла)

В директории `/train` располагаются два файла: `train.cpp` и `trench.cpp`.
В первом через `extern` реализовано использование функции из второго файла.

Ваша задача заключается в том, чтобы создать target **train**, в рамках которого будет создан исполняемый файл `train`, который будет лежать в папке `/results`.
Папка должна быть создана во время сборки, в самом проекте ее нет.

Данный этап будет проверяться следующими коммандами:

```shell
w@double:~/task-cmake$ cmake -S . -B build
w@double:~/task-cmake$ cmake --build build --target train
w@double:~/task-cmake$ cd results && ./train 11
```

## Этап 2 (6.5 баллов)

Теперь же нужно полностью реализовать сборку проекта.

Устройство проекта следующее:

1. В папке `/drachenland` есть три поддиректории:
    * `fiery` и `icy` - тут лежат файлы вида `[..]_dragon.cpp` вместе с одноименными хедерами.
    Каждому такому файлу для работы надо подключить `breath_power.hpp` из папки `/utils`.
    * `fury` - тут располагаются файлы `toothless.c` и `toothless.h`, написанные на языке **C**, им никакие зависимости (в смысле `#include`) не требуются.
2. В папке `/utils` располагаются:
    * `breath_power.hpp` - хедер, который потом добавляют в другие файлы
    * `includeLint.py` - линтер директив `#include`. Он имеет флаг `-d`, после которого передается путь до директории, которую необходимо проверить (скрипт рекурсивно обходит все поддиректории)
   Он проверяет все `.cpp` и `.hpp` файлы на наличие относительных путей в директивах.
   Если такие были обнаружены, то скрипт завершается с ошибкой
3. В папке `/arena` располагаются тесты, в рамках которых происходят поединки драконов.
   * `.hpp` файлы подключены через простой `#include`
   * `.h` файлы подключены через `extern "C" { #include }`

Итак, вам нужно:

1. Сделать так, чтобы можно было делать `#include` без относительных путей (сам код уже настроен)
2. Добавить target **lint**, в рамках которого будет происходить запуск линтера для корневой директории проекта.
Данный target должен сопровождаться комментарием `Executing linter`. Сделайте так, чтобы этот target исполнялся всегда (подсказка: *прочитайте документацию CMake - вам нужно будет добавить специальную опцию*).
3. Установить компиляторами gcc и g++ (без этого не пройдет сбор покрытия)
4. Добавить в сборку тесты

> Вам надо будет подключить gtest для тестирования, который требует для выполнения `-pthread`:
>
> "We need `-pthread` as Google Test uses threads." @googletest/README.md

5. Собрать покрытие тестами

> Для этого мы предлагаем вам использовать сразу готовый [модуль](https://github.com/libevent/libevent/blob/master/cmake/CodeCoverage.cmake).
> В его описании указано, как использовать предоставляемые им функции.
> 
> Но не все так просто. Давайте сделаем динамическую выгрузку этого модуля сразу в папку `/cmake`.
> Для пободного действия предлагаем воспользоваться утилитой `wget`. Полную документацию можно найти на [официальном сайте GNU](https://www.gnu.org/software/wget/manual/wget.html).
> Однако она избыточна для наших целей, поэтому всю необходимую информацию с примерами можно найти [тут](https://losst.pro/komanda-wget-linux).
> 
> Осталось лишь найти url, по которому можно скачать файл. Тут все просто: github предоставляет такую возможность.
> Для это нажмите на кнопку `Raw` в правом верхнем углу, после чего в адресной строке появится необходимый url.
> 
> ![img.png](raw_gh.png)
> 
> * target сборки покрытия назовите **coverage**
> * директория с результатами покрытия - `/coverage`
> 

6. Добавить поддержку двух профилей(вариантов) сборки:
   1. **Дневные поединки**. Запуск будет осуществляться с опцией `-DDAY=ON`. В рамках данной сборки компилятор должен получить опцию `-Dday`
   2. **Ночные поединки**. Аналогично, но теперь опции запуска и компилятора будут соответственно `-DNIGHT=ON` и `-Dnight`

Проверка этого этапа будет проходить следующим образом:

1. Проверка тестов (2.5 балла)

```shell
w@double:~/task-cmake$ cmake -S . -B build -DDAY=ON
w@double:~/task-cmake$ cmake --build build
w@double:~/task-cmake$ cd build && make test
```

```shell
w@double:~/task-cmake$ cmake -S . -B build -DNIGHT=ON
w@double:~/task-cmake$ cmake --build build
w@double:~/task-cmake$ cd build && make test
```

2. Проверка сборки покрытия (4 балла)

```shell
w@double:~/task-cmake$ cmake -S . -B build -DDAY=ON
w@double:~/task-cmake$ cmake --build build --target coverage
```

## Этап 3 (2 балла)

Теперь вам предстоит реализовать target **clear**, в рамках которого будет происходить очистка результатов сборки. А именно следует удалить:
* Папку сборки (она будет передана через опцию `-DBUILD_SOURCE_DIR=[..]` - путь будет относительно корневой директории проекта, т.е. `test-cmake`)
* Файл `/cmake/CodeCoverage.cmake`, который был динамически скачан
* Директорию с результатами покрытия `/coverage`
* Директорию с результатами сборки бинарника из первого этапа `/results`

> Добавьте комментарии к этой стадии, описывающие действия

Проверка будет осуществляться следующим образом:

Первый случай

```shell
w@double:~/task-cmake$ cmake -S . -B build -DDAY=ON -DBUILD_SOURCE_DIR="build"
w@double:~/task-cmake$ cmake --build build --target clear
```

Второй случай
```shell
w@double:~/task-cmake$ cmake -S . -B build -DDAY=ON -DBUILD_SOURCE_DIR="build"
w@double:~/task-cmake$ cmake --build build --target train
w@double:~/task-cmake$ cmake --build build --target coverage
w@double:~/task-cmake$ cmake --build build --target clear
```

> Обратите внимание, что в первом случае папка `results` не будет существовать.

> За каждый из случаев дается 1 балл

---

Немного про зависимость баллов:

* Если этап 1 не пройден, то далее тесты не исполняются
* Если на каком-то этап `cmake` отрабатывает с ошибками, то исполнение тестов прекращается
* Если не выполнено задание по сборке покрытия тестов, то этап 3 не тестируется
* Если нет стадии линтера, то далее тесты не выполняются
